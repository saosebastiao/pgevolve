# pgevolve v0.2 — Architectural Review for Readiness

- Status: draft, awaiting review (revision 2)
- Date: 2026-05-15
- Authors: Daniel Toone
- Scope: Pre-v0.2 architectural review. Audits which components scale
  cleanly to the planned v0.2 surface and which need to evolve.
- Sibling spec:
  [`2026-05-15-test-strategy-v2-design.md`](./2026-05-15-test-strategy-v2-design.md)
  consumes the conclusions here.

## 1. Motivation

v0.1 ships with full table / index / sequence / constraint / schema /
comment support. The v0.2 roadmap (per `docs/spec/objects.md`) adds:

- Views and materialized views (sub-spec drafted; needs revision against
  this review).
- Functions and procedures (SQL + PL/pgSQL bodies).
- Triggers (row- and statement-level; constraint triggers).
- User-defined types: enums, domains, composites.
- Extensions (`CREATE EXTENSION` with version pins).
- Declarative partitioning (range / list / hash).
- Table reloptions (`fillfactor`, autovacuum, toast).

Each new family stresses different architectural seams. The first draft
of this review chose shadow PG as the canonical body normalizer, which
created a bootstrap circularity (we need an order to apply to shadow,
which we'd be using shadow to derive) and a hard Docker dependency.
This revision pivots: **the AST is the source of truth for dependency
extraction and ordering; shadow PG becomes an opt-in cross-check.** The
spec produces decisions, not implementation. Each decision lands as a
`Decision N` callout with the rationale; the v0.2 sub-specs and the
sibling test-strategy spec are the implementation venues.

## 2. Non-goals

- Reopening v0.1 invariants that already hold under the new surface
  (canonicalization is byte-stable, plans are reviewable as SQL, no
  silent data loss).
- Designing the v0.2 sub-specs that this audit unblocks. Sub-spec drafts
  remain the home for object-kind-specific decisions (e.g., which PG
  catalog tables the function reader queries).
- Building a generic plugin loader. See §11 for why we resist that.
- Cluster-level concerns (`postgresql.conf`, roles, GRANT) in v0.2.
  Sketched in §17 as the v0.3 surface but not designed here.

## 3. Method

For each component in `docs/system/architecture.md` we ask three
questions against each upcoming v0.2 object family:

1. Does the existing abstraction accept the new family without change?
2. If not, is the smallest extension a **new variant** in an existing
   enum, a **new collection** on `Catalog`, or a genuinely **new
   mechanism**?
3. Where does state live — in the IR (a fact about the desired schema),
   in the dep graph (a fact about ordering), in the planner policy (a
   fact about how to apply the change), or in lint (a fact about
   reviewability)?

The answers populate §5's capability matrix and the decisions in §6–§12.

partial — `EXECUTE`-style dynamic SQL is opaque to any AST. The
mechanism handles SQL-language bodies and PL/pgSQL static SQL fully;
dynamic SQL is closed by user directives (Decision 11) and the
optional shadow cross-check (Decision 12).

The matrix highlights three concentrations of genuinely new mechanism:

1. **AST-driven body and dep extraction** (views first, then reused
   across functions, triggers, expression indexes, generated columns,
   check constraints).
2. **Dep graph extended with body-derived edges and signature-keyed
   nodes** (functions; required for triggers and expression indexes).
3. **Executor transaction patterns for `ALTER TYPE ADD VALUE`** (enums)
   and for online enum migration semantics.

The rest is additive variants over existing abstractions. The v0.1
architecture holds.

## 6. IR shape

### Decision 1 — IR stays flat, gains seven new top-level collections.

`Catalog` becomes:

```rust
pub struct Catalog {
    pub schemas:            Vec<Schema>,
    pub extensions:         Vec<Extension>,           // new
    pub types:              Vec<UserType>,            // new (enum / domain / composite)
    pub tables:             Vec<Table>,
    pub indexes:            Vec<Index>,
    pub sequences:          Vec<Sequence>,
    pub views:              Vec<View>,                // new (views spec §4)
    pub materialized_views: Vec<MaterializedView>,    // new (views spec §4)
    pub functions:          Vec<Function>,            // new
    pub procedures:         Vec<Procedure>,           // new
    pub triggers:           Vec<Trigger>,             // new
}
```

Rationale: the flat-collections shape has paid for itself in v0.1. Pair-
by-key diff stays trivially correct; canonicalization stays linear;
serialization for `PlanId` hashing stays byte-stable. Nesting would
either force every consumer to walk a tree or force us to maintain
referential integrity at every mutation. We have a precedent (`Index`
already lives flat with a `table: QualifiedName` reference) — repeat
the pattern.

### Decision 2 — Identity keys: qname + optional disambiguator.

| Object kind | Identity |
|---|---|
| Schema, table, sequence, view, MV, index, trigger, procedure, enum, domain, composite, extension | `QualifiedName` |
| Function | `(QualifiedName, NormalizedArgTypes)` |

Function overloads share a qname but differ by argument types. Postgres
catalog uniqueness is `(pronamespace, proname, proargtypes)`; pgevolve
mirrors that exactly. `NormalizedArgTypes` is `Vec<ColumnType>` after
running the existing `ColumnType::parse_from_pg_type_string`. This is
the only kind in v0.2 that needs a disambiguator.

### Decision 3 — Bodies live in canonical text plus structured dep edges.

Every body-bearing IR record (`View`, `MaterializedView`, `Function`,
`Procedure`, `Trigger`, expression-`Index`, generated-`Column`,
check-`Constraint`) carries:

```rust
struct Body {
    canonical_text: String,             // pgevolve-side AST-canonicalized form
    canonical_hash: [u8; 32],           // BLAKE3 of canonical_text; fast equality
    dependencies:   Vec<DepEdge>,       // AST-extracted; column-level when applicable
}
```

(The existing `NormalizedExpr { canonical_text, ast_hash }` pattern
generalizes to this. The new `NormalizedBody` type is the
statement-scope counterpart.) Diff compares `canonical_hash`
byte-equal. Body diff is "replace" — there is no partial-body diff.

The canonicalizer is pgevolve-side, not PG-side. Source and catalog
both produce the same canonical form by walking the same AST shape
through `NormalizedBody::canonicalize`. The catalog reader passes
`pg_get_*` output through the canonicalizer to land in the same shape.
This eliminates the bootstrap circularity of needing a working PG to
canonicalize the bodies we'd then apply to that PG.

### Decision 4 — Transient states are not desired-state facts; catalog reader exposes them as observation-side drift kinds.

The existing invariant — "the IR represents only fully-validated
state" — is preserved for *source* IR. But the catalog reader must
detect and report the transient states that real production databases
fall into:

| Observed state | Source of truth | Drift kind | Planner response |
|---|---|---|---|
| `NOT VALID` constraint | `pg_constraint.convalidated = false` | `ConstraintPendingValidation` | Emit `VALIDATE CONSTRAINT`. |
| `INVALID` index | `pg_index.indisvalid = false` | `IndexInvalid` | Emit `REINDEX` or `DROP INDEX + CREATE INDEX` per policy. |
| MV `WITH NO DATA` | `pg_class.relispopulated = false` | `MvUnpopulated` | Emit `REFRESH MATERIALIZED VIEW`. |
| Enum value uncommitted (PG ≤ 11, mid-`ALTER TYPE ADD VALUE`) | `pg_enum.enumsortorder` anomaly | `EnumValueUncommitted` | Emit reorder + re-add. |

Source IR represents only the validated end-state. Catalog reader
distinguishes "constraint exists and validated" from "constraint exists
but pending validation." Diff produces a resolution change rather than
a false add or a silent miss. This makes partial-apply recovery
automatic: a crash between `ADD CONSTRAINT NOT VALID` and `VALIDATE
CONSTRAINT` no longer leaves the database in a state pgevolve can't
re-plan from.

Permanently-not-valid constraints (the legitimate use case where a
team explicitly wants enforcement on new rows only) are a separate
desired-state feature. Deferred to v0.3; tracked in §14 as an open
question.

## 7. Dependency graph

### Decision 5 — Two edge sources: name-derived and AST-derived; unioned.

v0.1 derives every edge from structural IR facts (FKs, sequence
ownership, etc.). Body-bearing objects in v0.2 cannot. The edge sources
are now:

- **Name-derived edges (existing).** `schema ← table`, `table ←
  index`, `table ← FK constraint → table`, `sequence ← owning table`.
- **AST-derived edges (new).** `body-bearing object ← every object it
  references, column-level when applicable`. Produced by walking the
  `pg_query` AST for the body and recording every relation, function,
  type, and column reference.

`order::build_graph` takes the union. Topo sort is unchanged in shape.
Three-bucket (creates / modifies / drops) split is unchanged.

### Decision 6 — AST extraction is the canonical dep source. Shadow PG is an optional cross-check, not a dep source.

The previous draft had body-derived edges resolving through shadow's
`pg_depend`. That created a bootstrap circularity: we need a deploy
order to apply to shadow, which we'd be using shadow to derive. AST
extraction breaks the cycle.

Concretely, source loading runs in this order:

1. **Parse pass.** `pg_query` parses every `*.sql` file into ASTs.
2. **Structural pass.** Build the provisional IR from each top-level
   statement (existing v0.1 logic).
3. **AST resolution pass (new).** For each body-bearing record, walk
   the body AST and produce `DepEdge`s. Resolve each reference against
   the provisional IR. Unresolved references (view body references
   missing column) become **AST resolution errors** with source
   location, before any DB touch.
4. **Canonicalization pass.** Each body's `canonical_text` is produced
   by re-rendering the resolved AST through `NormalizedBody::canonicalize`.
5. **Final IR canonicalization.** Sort, dedupe (existing v0.1 logic).

The result: pgevolve behaves like a compiler. Errors point at source
lines, not at shadow boot logs. The whole pipeline is reproducible
without a running Postgres.

### Decision 7 — Function signature in `NodeId`.

`NodeId` for functions is `(QualifiedName, NormalizedArgTypes)`.
`Ord` on the new pair is lexicographic on (qname, args). The min-heap
tiebreaker in Kahn's algorithm continues to produce deterministic
order.

### Decision 8 — Body-derived cycles surface as errors, not auto-broken.

FK cycle extraction (the only auto-cycle-break in v0.1) is well-defined
because every cycle node is `Table` / `Constraint` and the fix is
mechanical. Body-derived cycles between views, or between functions and
views, have no general mechanical fix (PG itself can't create them
inline). When `order` detects a body-derived cycle, it returns
`PlanError::BodyCycle { nodes }` listing the cycle. The user fixes the
source.

## 8. AST-derived dependencies and optional shadow validation

### Decision 9 — AST-derived dep graph is the source of truth; computed before any DB touch.

The full dep graph (name-derived + AST-derived) is computed at source
load time. This unlocks:

- **Compiler-like diagnostics.** "View `app.summary` references column
  `users.legacy_id` which does not exist (defined at `users/users.sql:42`,
  referenced at `views/summary.sql:7`)." No more opaque shadow-boot
  failures.
- **Reproducible plans without containers.** Plans are byte-stable from
  source alone. Docker is no longer required for the normal `plan`
  hot path.
- **Pre-flight verification.** The catalog reader's drift check can
  run against the AST-derived dep graph to catch "production has an
  object referencing something we no longer declare" before apply.

### Decision 10 — Body canonicalization is pgevolve-side; `NormalizedBody` extends `NormalizedExpr`.

`NormalizedExpr` already handles expression-scope canonicalization (paren
folding, keyword case, type-alias normalization, commutative operand
sort, redundant-cast stripping). The same primitives extend to
statement-scope bodies. `NormalizedBody` is a newtype around a
canonical-text string plus a BLAKE3 hash; its constructor walks the AST
and emits the canonical form.

Source-side: the parser produces `NormalizedBody` from the parsed AST.
Catalog-side: `pg_get_viewdef`, `pg_get_functiondef`, etc. produce
text that we re-parse with `pg_query` and pass through the same
canonicalizer. Byte-equality on `canonical_text` is the diff predicate.

Maintaining the canonicalizer is real work — every new PG syntax form
(`MERGE`, new operator classes, new function attributes) needs a
canonicalization rule. The maintenance surface is bounded and grows
linearly with PG release notes; we already maintain
`ColumnType::parse_from_pg_type_string` at this granularity.

### Decision 11 — PL/pgSQL dynamic-SQL deps are partial; user directives close the gap.

Static SQL inside PL/pgSQL bodies (`SELECT INTO`, embedded queries) is
parseable and yields full dep edges. Dynamic SQL (`EXECUTE format(...)`)
is opaque to any AST.

For each function whose AST resolution flags any opaque region, the
linter emits `pl-pgsql-dynamic-sql` pointing at the line. Users close
the gap with explicit directives in source SQL comments:

```sql
CREATE OR REPLACE FUNCTION app.refresh_summary() RETURNS void
LANGUAGE plpgsql AS $$
  -- @pgevolve dep: app.summary
  -- @pgevolve dep: app.users.email
BEGIN
  EXECUTE format('REFRESH MATERIALIZED VIEW %I.summary', current_schema());
END;
$$;
```

Directives are first-class IR — `DepEdge::source = DepSource::AstDeclared`
vs `DepSource::AstExtracted`. Both flow through the same graph. The
lint clears once every flagged opaque region is matched by at least
one directive in the same function body.

### Decision 12 — `--shadow-validate` is opt-in; backend-pluggable.

A separate `pgevolve plan --shadow-validate` flag (also available on
`diff` and `validate`) boots a Postgres of the target major, applies
the plan, reads back `pg_depend` and the catalog-side bodies, and
cross-checks:

- Every AST-extracted edge appears in `pg_depend`.
- No `pg_depend` edge is missing from the AST extraction (unless
  covered by a user directive).
- Every body's `canonical_text` round-trips byte-equal after re-canonicalizing the catalog-side output.

Mismatches surface as warnings by default, errors under `--strict`.
Shadow-validate is the discovery loop for canonicalizer bugs and for
PL/pgSQL deps the user forgot to declare.

The Postgres instance used by shadow-validate is pluggable via the
`[shadow]` block in `pgevolve.toml`:

```toml
[shadow]
backend = "auto"     # "auto" | "testcontainers" | "dsn"
url     = "postgres://localhost/myapp_shadow"   # for backend = "dsn"
url_env = "PGEVOLVE_SHADOW_URL"
reset   = "drop_schema_cascade"                  # "drop_schema_cascade" | "drop_database_create" | "none"

# When backend = "testcontainers", `extensions` is installed in the
# fresh container before apply. When backend = "dsn", the user is
# responsible for pre-installing them.
extensions = ["pgcrypto", "uuid-ossp"]
```

`backend = "auto"` prefers a user-supplied DSN if configured, else
falls back to testcontainers if Docker is present, else errors with a
clear pointer to the config. This means:

- Devs without Docker can run `--shadow-validate` against a local PG
  or a per-developer ephemeral DB.
- Custom PG configurations (TimescaleDB, PL/v8, non-default GUCs) are
  supported by pre-configuring the user-supplied shadow.
- CI uses testcontainers for hermetic isolation.

The `ShadowBackend` trait that selects between modes is itself an
extension seam (per Decision 22): future backends (`pg_tmp`, embedded
PG via a Rust crate, managed-PG plugins) implement it without
changing core.

## 9. Failure-mode policy

### Decision 13 — Three explicit failure tiers, mutually exclusive.

| Tier | What fails | Detected at | User remedy |
|---|---|---|---|
| **Parse** | Syntax errors; duplicate qnames; unknown statement kinds; layout-profile violations (under strict mode). | Parser, before any resolution. | Fix source SQL. |
| **AST resolution** | Semantic errors detectable without a running PG: view body references missing column, function body references missing type, FK to nonexistent table, trigger function not found, body-derived dependency cycle. | AST resolution pass (Decision 6 step 3). | Fix source SQL. |
| **Lint at plan** | Drift we detect but can't act on automatically: column position drift, redundant index, missing extension on referenced type, dynamic-partition naming clashes. | Planner, after diff. | Acknowledge (lint waiver), fix source, or run the appropriate corrective CLI command. |

The previous draft named the middle tier "Shadow-load"; under the
AST-first pivot, that work happens deterministically at AST resolution
time without booting anything. Shadow-validate (Decision 12) is a
separate opt-in stage and does *not* gate plan generation by default.

No "build a plan with broken references" path exists. The current v0.1
silent treatment of column reorder is reclassified into the lint tier.

### Decision 14 — Lint-at-plan is hard-fail by default; waiverable via `intent.toml`.

A new `[[lint_waiver]]` row in `intent.toml`:

```toml
[[lint_waiver]]
rule   = "column-position-drift"
target = "app.users.email"
reason = "applied via separate rewrite-table operation; see PR #234"
```

Waiver rows are reviewed in PRs the same way `[[intent]]` rows are.
Without a waiver, plan fails with exit code 2 (the same code as drift
detection — semantically aligned).

### Decision 15 — `pgevolve diff` always succeeds; `plan` enforces the policy.

`diff` is a read-only inspection that surfaces the change set including
lint findings, but doesn't gate. `plan` enforces lint-at-plan failure.
This preserves `diff`'s usefulness as a debugging tool against any
state.

## 10. Column ordering and the rewrite-table escape hatch

### Decision 16 — Column position drift is lint-only by default.

PG has no `ALTER COLUMN ... POSITION`. Reordering requires a table
rewrite (`CREATE TABLE new ...; INSERT INTO new SELECT ...; DROP TABLE
old; RENAME`). That is expensive, destructive (in the
"requires-approval" sense; data preserved but downtime real), and
not the right default.

The lint rule `column-position-drift` fires when diff detects a column
position mismatch but no other column change. Users either:

- Reorder the source SQL to match the live DB (lint clears).
- Waive the lint and accept the drift indefinitely.
- Run a separate command (next decision) that authorizes the rewrite.

### Decision 17 — `pgevolve rewrite-table <qname>` as a separate command.

Not a planner step. A standalone CLI command that:

1. Confirms the target column order matches the source.
2. Builds a one-table migration plan: create shadow table, copy
   rows, swap names, drop old.
3. Requires an explicit `--confirm-rewrite` flag plus an
   `intent.toml` row.
4. Produces a regular plan directory so the rewrite is reviewable in
   PR.

Future use cases for the same command: online column-type change
(int → bigint), changing partition key, restructuring constraints into
PK swap. The command is the rewrite escape hatch for any operation PG
doesn't offer natively.

## 11. Dynamic objects (partman / TimescaleDB / pg_cron)

### Decision 18 — Per-table `managed_partition_scheme` setting; default `"static"`.

Two modes for v0.2:

- `static` (default): pgevolve fully manages partitions. Every
  partition appears as a `Table` with `partition_of: Some(parent)` in
  source; partitions not in source diff as drift.
- `ignore_runtime_children`: pgevolve manages the parent and the
  partitions declared in source, but ignores any partition matching a
  configurable `name_pattern` (default: tool-specific patterns for
  `partman_*`, `_hyper_%`, `_chunk_%`). Drift on those names is
  silently allowed.

Configured per-table in source SQL via a `pgevolve` comment directive,
or globally in `pgevolve.toml`:

```toml
[managed.partitions]
ignore_runtime_children_pattern = ["partman_*", "_hyper_*", "_chunk_*"]
```

### Decision 19 — Declarative partition policy is deferred (v0.4+).

A `partition_policy = { kind = "time", interval = "1 month",
retention_months = 12 }` declarative form is interesting but
out-of-scope for v0.2. The `ignore_runtime_children` mode gives users
a path to coexist with partman / TimescaleDB today without committing
us to reimplement them.

### Decision 20 — Extension management surfaces the prerequisite, not the workflow.

Extensions are first-class in v0.2 (per `docs/spec/objects.md`).
pgevolve declares the extension dependency via `[extensions]` in
`pgevolve.toml`; the extension manages its own runtime objects. We
do *not* manage objects owned by an extension (per the existing v0.2
extensions design line). Coexistence with partman / TimescaleDB
therefore reduces to: declare the extension, configure
`ignore_runtime_children`, write any user-owned partitions
declaratively.

## 12. CLI gaps

### Decision 21 — Add five commands in v0.2; defer a generic plugin loader.

| Command | Purpose | Touches DB | Writes files |
|---|---|---|---|
| `pgevolve baseline --db <env> -o <dir>` | Capture current state as authoritative source. Alias for `dump`; `dump` itself was already planned in v0.1.x. | read-only | yes |
| `pgevolve graph --format=dot\|mermaid` | Render the dep graph (name-derived + AST-derived edges) for current source or for a plan. Hugely useful when debugging "why did this view get recreated". Computed from AST alone; no DB or container required. | no | optionally to file |
| `pgevolve test <fixture-dir>` | Run a conformance-style before/after fixture through the local pipeline and print each assertion layer. Same harness the suite uses. | optional `--shadow-validate` | no |
| `pgevolve doctor --db <env>` | Project health: bootstrap status, drift state, dangling intents, abandoned plans, extension version drift, advisory-lock holders, recent apply failures, GUC mismatches between source assumptions and live DB, ignored-by-`ignore_runtime_children` object counts. | read-only | no |
| `pgevolve rewrite-table <qname> --confirm-rewrite` | The rewrite escape hatch (Decision 17). | read+write | yes |

`refresh-mv` is *not* a separate command: `pgevolve apply` of a plan
containing the `refresh_materialized_view` step kind is the canonical
path. Out-of-band refresh is a DBA action, not a pgevolve action.

### Decision 22 — Resist a generic plugin / hook system. Document existing seams.

Today's extension seams cover the foreseeable surface:

- `[planner.online_rewrites]` toggles for new rewrites.
- `layout_profile` mechanism for project-specific source layout rules.
- `CatalogQuerier` trait for alternate backends.
- `ShadowBackend` trait (new, per Decision 12) for shadow PG providers.
- Custom lint profiles (regex + assertion).

A generic plugin loader is premature. When a second consumer asks for
a hook that none of these seams covers, we revisit. Document the five
seams above as the public extension API in `docs/spec/`.

## 13. Per-component implications summary

The decisions above translate into per-component work. The v0.2
sub-specs are the implementation venues; this summary exists so each
sub-spec author knows which architectural decision they're standing
on.

- **Parser.** New `*Stmt` handlers per family. New AST resolution pass
  (Decision 6 step 3) that walks bodies and produces `DepEdge`s and
  `NormalizedBody`s. Captures source locations for every dep edge so
  diagnostics point at the right line.
- **Catalog reader.** Per-family `read_*` functions and per-PG-major
  SQL strings. Returns `Body` populated by passing `pg_get_*` output
  through the same `NormalizedBody::canonicalize`. Detects transient
  states per Decision 4 and emits the corresponding drift kinds.
- **IR.** Seven new top-level collections (§6). `Body` shape factored
  out of `NormalizedExpr` as `NormalizedBody`. Function `NodeId`
  carries signature (§6, §7).
- **Diff.** New `Change` variants per family; pair-by-key extended to
  function-signature keys; body comparison is hash-equal. New drift
  kinds from Decision 4.
- **Order.** Two edge sources unioned (§7). `PlanError::BodyCycle`
  added. AST resolution failures (Decision 13 middle tier) surface
  here.
- **Rewrite.** New step kinds per family (per the views sub-spec
  pattern). One new mechanism: enum `ALTER TYPE ADD VALUE` outside
  transactions on PG ≤ 11.
- **Group.** Mostly unchanged. Enum-value-add is its own
  non-transactional group on PG ≤ 11; honored by existing transactional-
  constraint flag.
- **Plan format.** `[[lint_waiver]]` rows in `intent.toml` (§9).
  `[[intent]]` shape unchanged. Manifest no longer carries a
  `shadow_cache_id` (the previous draft assumed shadow was canonical;
  it isn't). Manifest gains `ast_resolution_version` so plan
  reproducibility is gated on canonicalizer version.
- **Lint.** New rules per family; `column-position-drift` reclassified
  to lint-at-plan; lint waivers honored via `intent.toml`; new
  `pl-pgsql-dynamic-sql` rule (Decision 11).
- **Executor.** New transactional grouping for enum value add; otherwise
  no structural change. Per-step audit unchanged.
- **Shadow.** Repurposed from "validate sidekick" to "optional
  cross-check backend" (Decision 12). New `ShadowBackend` trait;
  testcontainers and DSN-backed implementations; auto-select policy.
- **Testkit.** IR generator + mutator gain coverage of the new
  families. `MigrationFixture` shape unchanged. New `TestPgBackend`
  trait mirrors `ShadowBackend` for the test side (sibling test-strategy
  spec §9).
- **Conformance.** New per-family fixture subtrees. Assertion layers
  unchanged in shape; sibling test-strategy spec defines new layers
  (minimality, no-collateral-damage, intent-shape, dep-graph,
  topological-order).
- **CLI.** Five new commands (Decision 21). Existing commands gain
  `--shadow-validate` flag where applicable.

## 14. Open questions

- **Procedure transactional semantics.** Procedures can `COMMIT` /
  `ROLLBACK` mid-body, which fights pgevolve's per-step transaction
  model. v0.2 procedure sub-spec should decide whether procedures
  with `COMMIT` in the body are rejected at lint time or run
  outside-transaction at apply time. Architectural impact: minimal
  (one new flag on a step kind). Decision deferred to the sub-spec.
- **Extension uninstall and dependent-object lifecycle.** When an
  extension is removed from source, every object the extension owned
  vanishes from the catalog after `DROP EXTENSION CASCADE`. Should
  pgevolve issue `CASCADE`, or refuse and require explicit removal of
  dependents? Lean: refuse, surface as lint, require explicit
  `[[intent]]` row for `DROP EXTENSION CASCADE`. Sub-spec decides.
- **Body canonical form across PG patch versions.** Some PG patch
  versions tweak `pg_get_functiondef` formatting. Because the
  canonicalizer is pgevolve-side and runs on the AST (not the text),
  patch-version drift is largely absorbed. Open: do we re-bless
  per-version fixtures on PG patch upgrades, or call our
  canonicalizer the contract and treat any PG drift as a PG bug?
  Test-strategy spec handles this.
- **Permanently-not-valid constraints as desired state.** Decision 4
  covers transient `NOT VALID` as drift; the legitimate use case of
  "enforce on new rows only, never validate against history" needs a
  source-side declaration syntax. Deferred to v0.3 unless a v0.2 user
  asks.

## 15. Risks

- **AST canonicalizer maintenance.** Every new PG syntax form needs
  a canonicalization rule. Mitigation: the per-PG-major delta docs
  (`docs/spec/pg-versions/pg<N>.md`, from the C-suite design) include
  a "canonicalizer impact" subsection. A new syntax that affects
  canonicalization blocks PG-version support until handled.
- **PL/pgSQL dep coverage gaps slip past review.** Users may not add
  the required `@pgevolve dep:` directives; the lint catches the
  *presence* of dynamic SQL but can't verify the directive set is
  complete. Mitigation: `--shadow-validate` (Decision 12) compares
  AST edges against `pg_depend`; any mismatch surfaces an error.
  Document the workflow as "AST first, --shadow-validate before
  merge."
- **IR file size for projects with many functions.** PL/pgSQL bodies
  can be long. `Catalog` is bincode-serialized for `PlanId` hashing;
  manifest holds JSON. Mitigation: `body_canonical` strings are
  deduplicated by `canonical_hash` in the manifest (a future
  optimization; not required for v0.2 correctness).
- **Footgun: `ignore_runtime_children` swallowing real drift.** If a
  user mistypes the partman pattern, drift on legitimate partitions
  goes silent. Mitigation: `pgevolve doctor` reports counts of
  ignored objects; users see them and can audit.
- **GUC mismatch between source assumptions and live DB.** Pgevolve
  canonicalizes assuming PG defaults; a database with
  `standard_conforming_strings = off` may produce drift the user
  can't explain. Mitigation: `pgevolve doctor` surfaces GUC-mismatch
  warnings.

## 17. Cluster-level surface (v0.3 preview)

This spec scopes cluster-level state out of v0.2 but commits the
architectural shape so users have a roadmap and v0.3 sub-specs have a
starting point.

### Decision 23 — Cluster state is a separate project, separate command surface, separate executor.

A pgevolve project today owns one database. Cluster state — roles,
GRANTs, tablespaces, `postgresql.conf` GUCs, database list — has a
different lifecycle, different audit needs, different credentials
(superuser, not DB owner). Modeling it inside per-DB projects would
force every plan to require superuser. Wrong default.

Sketch:

```
my-cluster/
  pgevolve-cluster.toml      # cluster-wide config + roles + tablespaces
  roles/                     # role definitions, source-of-truth
  grants/                    # GRANT statements organized by role
  databases/                 # one entry per DB the cluster hosts (with link to its own pgevolve.toml)
```

CLI surface (preview, v0.3):

- `pgevolve cluster init` / `lint` / `diff` / `plan` / `apply` /
  `status` — sibling to the per-DB commands, run with superuser DSN.
- Per-DB `pgevolve apply` does not touch cluster state. Cluster
  apply does not touch per-DB schema state.
- Cluster IR: `ClusterCatalog { databases, roles, tablespaces,
  cluster_settings, foreign_servers, user_mappings }`. Same IR shape
  pattern (flat collections + pair-by-name diff).
- Cluster plan format: same `plan.sql + intent.toml + manifest.toml`
  shape, in a separate `cluster-plans/` directory.

For v0.2, users handle cluster state outside pgevolve (Terraform,
Ansible, hand-rolled SQL). The v0.3 sub-spec is the design venue for
the full surface.
